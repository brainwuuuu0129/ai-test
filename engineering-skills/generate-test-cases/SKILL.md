# 测试用例生成 (Generate Test Cases)

## 🚨 输出格式强制要求（最高优先级）

**必须输出为 Lark 脑图导入格式的 `.txt` 文件（Tab 缩进纯文本）。**

- **禁止** 输出 Markdown 表格格式
- **禁止** 输出 `| 端 | 模块 | 测试点 |` 这种表格
- **唯一允许** 的格式是 Tab 缩进纯文本，可直接复制粘贴到 Lark 思维笔记
- 文件命名规范：`{功能名称} test case - lark mindnote.txt`

## 1️⃣ Purpose & Scope

将 PRD 和技术设计文档转化为**结构化、可执行、可导入 Lark 脑图**的测试用例。

- 覆盖所有功能点和业务场景
- 包含前端、后端、多端适配、埋点验证
- 输出格式：**Lark 脑图导入格式 `.txt`（Tab 缩进纯文本）**

---

## 2️⃣ Pre-Requisites

开始前需要准备：

| 输入 | 说明 | 必需 |
|------|------|------|
| PRD 文档 | 产品需求文档（Lark Wiki URL 或本地文件） | ✅ |
| 技术设计文档 | 开发设计方案（可选但推荐） | 推荐 |
| 已有测试用例 | 参考格式和思路（.mm 文件或 Lark mindnote） | 推荐 |

## 3️⃣ 执行流程

### Step 1: 读取 PRD 文档

**目标**：理解业务需求，梳理功能点

**操作**：
- 使用 Lark MCP 工具读取 PRD（`wiki_v2_space_getNode` + `docx_v1_document_rawContent`）
- 或读取本地 PRD 文件

**输出**：
- 功能模块清单
- 用户角色识别
- 核心业务流程
- 边界场景识别

### Step 2: 读取技术设计文档

**目标**：深入理解技术实现，挖掘后端测试点

**操作**：
- 读取技术设计文档
- 关注：数据库表结构、缓存策略、接口定义、触发机制

**输出**：
- 后端测试点（数据一致性、缓存命中、并发处理）
- 接口测试点
- 性能边界点

### Step 3: 参考已有测试用例

**目标**：学习团队测试用例的格式和思路

**操作**：
- 读取 .mm 格式的测试用例文件
- 分析层级结构、命名规范、前置条件设置方式

**学习要点**：
- 层级划分（平台 → 功能 → 场景 → 测试点）
- 前置条件的详细程度
- 操作步骤的颗粒度
- 预期结果的可验证性
- p0 标记的使用场景

### Step 4: 梳理测试重点

**目标**：确定测试覆盖范围和优先级

**输出清单**：
```
1. 前端测试点
   - UI 展示规则
   - 交互逻辑
   - 多端适配（PC/H5/App）
   - 响应式布局

2. 后端测试点
   - 数据统计逻辑
   - 缓存策略
   - 触发更新机制
   - 并发/边界处理

3. 业务规则测试点
   - 显示/隐藏规则
   - 数量约束
   - 状态联动

4. 埋点验证
   - 事件类型
   - 上报接口

5. 边界场景
   - 数据为空
   - 数据超限
   - 异常恢复

6. 流程图覆盖（如有）
   - 每个决策节点
   - 每个分支路径
   - 完整正向路径
   - 异常处理路径

7. 金额计算（如涉及）
   - 计算公式验证
   - 精度处理
   - 组合支付

8. 单元测试计划覆盖（如有）
   - 技术文档中列出的测试用例
```

### Step 4.1: 流程图覆盖分析（新增）

**目标**：根据技术文档中的流程图，确保每个节点和分支都有测试覆盖

**适用场景**：技术文档包含 Flowchart / 流程图

**分析方法**：

1. **识别流程图**：从技术文档中提取所有流程图（如：验证流程、计算流程、选择流程）

2. **节点拆解**：将每个流程拆解为节点
   - 开始节点
   - 决策节点（菱形，有分支）
   - 处理节点（矩形）
   - 结束节点

3. **分支覆盖**：每个决策节点的每个分支都需要测试用例
   ```
   决策节点：优惠券是否存在？
   ├── 是 → 进入下一步检查
   └── 否 → 返回 null
   ```

4. **路径覆盖**：
   - 完整正向路径（从开始到成功结束）
   - 各种失败路径（每个可能失败的节点）
   - 异常处理路径

**输出格式**：
```
流程图覆盖测试
	流程名称（方法名）
		节点1：[节点描述]
			场景A
				前置条件：xxx
					操作步骤：xxx
						预期结果：xxx
							p0
			场景B
				...
		节点2：[节点描述]
			...
		完整路径 - 成功
			前置条件：满足所有条件
				操作步骤：完整流程
					预期结果：流程成功完成
						p0
```

### Step 4.2: 金额计算全覆盖（新增）

**目标**：确保涉及金额计算的功能有完整的测试覆盖

**适用场景**：订单、支付、优惠券、退款等涉及金额的功能

**参考文档**：金额计算经验文档（如有）

**覆盖维度**：

1. **计算公式验证**
   ```
   - 基础公式：Total = Items - Discount + Tax + Delivery + Service
   - 折扣计算：Percentage / Fixed Amount / maxDiscount 限制
   - 分摊计算：多 items 按比例分摊
   ```

2. **Tax 计算**
   ```
   - Tax 基数：折后金额
   - Tax 精度：2位小数，HALF_UP
   - 不同地区税率
   ```

3. **Service Fee / Delivery Fee**
   ```
   - 与 Plan 关系
   - 与订单金额关系
   - 免费条件
   ```

4. **多 Items 分摊**
   ```
   - 按 subtotal 比例分摊
   - 精度处理（HALF_UP）
   - 最后一个 item 用 remain 避免累积误差
   ```

5. **组合支付**
   ```
   - Allowance + Self-Pay
   - WeBucks 抵扣
   - Rollover 使用
   - 支付优先级
   ```

6. **多支付渠道**
   ```
   - Stripe 支付
   - Alipay 支付
   - 支付方式切换
   - 汇率处理（如适用）
   ```

7. **退款计算**
   ```
   - 全额退款
   - 部分退款（按 item）
   - Tax 重算
   - 原路退款
   ```

8. **金额一致性**
   ```
   - 购物车 → Checkout → 支付页 一致
   - 前端 → 后端 一致
   - 订单详情 → 账单 一致
   ```

9. **边界值**
   ```
   - 极小金额（$0.01）
   - 零金额
   - 负金额防护
   - 大额订单
   - 精度（2位小数）
   ```

### Step 4.3: 单元测试计划覆盖检查（新增）

**目标**：确保技术文档中的单元测试计划都有对应的功能测试用例

**操作**：
1. 从技术文档中提取"单元测试计划"章节
2. 逐条对比当前测试用例
3. 补充遗漏的测试点

**示例**：
```
技术文档单元测试计划：
- testStartDateNotSet → 对应测试用例：startDate 为 null
- testMaxDiscountExceeded → 对应测试用例：折扣金额 > maxDiscountAmount
- testCompleteOrderFlow → 对应测试用例：完整下单流程（集成测试）
```

## 4️⃣ 输出格式规范（唯一格式：Lark 脑图 Tab 缩进纯文本）

### 文件规范

- **文件类型**：`.txt`（UTF-8 编码）
- **文件命名**：`{功能名称} test case - lark mindnote.txt`
- **缩进方式**：必须使用 **真实 Tab 字符**（`\t`），不可用空格代替
- **输出位置**：项目根目录（与其他 test case .txt 文件同级）

### 层级结构

```
功能名称 Test Case                          ← 第1层：根节点
	Website                                  ← 第2层：平台/端（Website / App / Guest Order / 后端接口 等）
		功能模块名称                            ← 第3层：功能模块
			测试场景                            ← 第4层：测试场景/分类
				测试点名称                       ← 第5层：具体测试点
					前置条件：xxx               ← 第6层：前置条件（可选，无则直接写操作步骤）
						操作步骤：xxx            ← 第7层：操作步骤
							预期结果：xxx        ← 第8层：预期结果
								p0              ← 第9层：优先级标记（仅核心用例标记 p0）
							预期结果：xxx        ← 同级多个预期结果
							预期结果：xxx
					前置条件：xxx               ← 同一测试点不同前置条件 = 不同场景
						操作步骤：xxx
							预期结果：xxx
```

### 格式规则

1. **前置条件**：以 `前置条件：` 开头，描述测试前的数据/状态/环境
2. **操作步骤**：以 `操作步骤：` 开头，仅使用动作动词（点击/输入/查看/选择/打开/进入/切换/等待）
3. **预期结果**：以 `预期结果：` 开头，必须是可观察的页面变化/状态变化/数据变化
4. **优先级**：核心用例在预期结果下一层标记 `p0`，非核心不标记
5. **多个预期结果**：同一操作步骤的多个预期结果放在同级（同缩进层级）
6. **无前置条件时**：测试点直接接操作步骤（跳过前置条件层级）

### 格式要点总结

| 要素 | 规则 | 错误示例 | 正确示例 |
|------|------|---------|---------|
| 缩进 | 必须用 Tab | 用4个空格缩进 | 用 Tab 缩进 |
| 前置条件 | `前置条件：` 开头 | `条件：用户已登录` | `前置条件：用户已登录` |
| 操作步骤 | `操作步骤：` 开头 | `步骤：点击按钮` | `操作步骤：点击按钮` |
| 预期结果 | `预期结果：` 开头 | `期望：显示弹窗` | `预期结果：显示弹窗` |
| 多预期结果 | 同级并列 | 写在一行 | 每个预期结果独占一行，同级缩进 |
| p0 标记 | 独占一行，缩进在预期结果下 | 写在预期结果同一行 | 预期结果下一行单独写 p0 |
| 平台分组 | 第2层按端分（Website/App/后端） | 不分端混写 | 按端分组 |

## 8️⃣ 使用示例

**用户输入**：
```
请根据这个 PRD 生成测试用例：https://xxx.larksuite.com/wiki/xxx
技术设计文档：https://xxx.larksuite.com/wiki/xxx
```

**执行流程**：
1. 读取 PRD → 梳理功能点
2. 读取技术文档 → 补充后端测试点
3. 生成 Lark 脑图导入格式 `.txt` 文件（Tab 缩进纯文本，可直接复制到 Lark 思维笔记）
4. 文件命名：`{功能名称} test case - lark mindnote.txt`
5. 输出到项目根目录

**绝对不要**：
- 不要输出 Markdown 表格格式（`| 端 | 模块 | 测试点 |`）
- 不要输出 `.md` 格式的测试用例
- 不要需要用户二次提醒格式

## 9️⃣ Key Principles

- **可执行性**：每个用例都能直接执行
- **可验证性**：预期结果可观察
- **完整覆盖**：功能点 + 后端逻辑 + 边界场景
- **格式统一**：遵循团队已有用例格式
- **优先级明确**：核心用例标记 p0
