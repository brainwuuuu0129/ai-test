# SKILL: 基于 PRD 和技术设计文档生成测试指南

## 概述

本技能用于根据以下输入，生成一份完整的软件模块**测试指南**文档：
- PRD（产品需求文档）
- 技术设计文档
- 现有测试用例（思维导图或结构化格式）

输出遵循标准化模板，适用于 QA 团队、回归测试及 AI Agent 知识库。

---

## 适用场景

- 用户需要为某模块生成测试指南、测试文档或测试 Wiki
- 用户提供 PRD、技术设计或测试用例链接（如飞书/Lark Wiki）
- 用户需要整合测试知识用于回归测试或工作交接

---

## 输入要求

| 输入 | 来源 | 用途 |
|------|------|------|
| **PRD** | 飞书 Wiki URL 或文档 | 提取业务背景、目标用户、业务目标、UI/UX 规格 |
| **技术设计** | 飞书 Wiki URL 或文档 | 提取系统架构、核心逻辑、边界场景、数据结构 |
| **测试用例** | 思维导图(.mm)、飞书 Wiki 或结构化文档 | 提取测试重点、测试方法、已知问题、回归清单 |
| **模板**（可选） | 飞书 Wiki 或用户提供 | 遵循特定的章节结构 |

---

## 输出结构（7 个章节）

### 1. 业务背景
- **概述**：模块的高层描述及其解决的问题
- **业务目标**：模块期望达成的关键成果（表格格式）
- **目标用户**：用户类型及其与模块的交互方式（表格格式）

### 2. 业务流程
- **流程图**：使用 Mermaid 绘制用户旅程、入口、操作和结果
- **关键操作**：按 PRD 列出阶段、操作和描述的表格
- **图片格式**：为不支持 Mermaid 的 Wiki 导出 PNG 流程图

### 3. 实现逻辑
- **3.1 系统架构**：组件、模块及其交互（含 ASCII 图）
- **3.2 核心逻辑**：业务规则、API 行为、展示逻辑、响应式规则、文案/Toast
- **3.3 边界场景**：每个边界场景的触发条件 + 处理方式

### 4. API
- **4.1 API 概述**：API 端点功能简述、第三方集成说明
- **4.2 端点详情**：URL、HTTP 方法、用途、请求/响应示例
- **4.3 错误处理**：常见错误状态码、含义及处理建议

### 5. 数据结构
- **5.1 数据库 Schema**：表、字段、关系（含图示）
- **5.2 数据校验**：按表/字段/API 的校验规则
- **5.3 示例数据**：关键实体的示例记录

### 6. 测试用例
- 测试用例文档链接
- **测试方法**：详细的回归测试结构，覆盖：
  - 按范围（页面/模块）
  - 按平台（PC/H5/APP）
  - 按用户/偏好状态
  - 按视口（响应式断点）
  - 可执行步骤（仅限点击、悬停、选择、等待、导航）
  - 可观察的预期结果
  - 回归清单

### 7. 典型问题
- 认证失败
- 数据处理错误
- API 错误
- UI/UX 缺陷
- 性能问题
- 并发问题

## 执行步骤

### 步骤 1：收集文档

```
1. 从用户获取 PRD URL
2. 调用 wiki_v2_space_getNode 获取文档 token
3. 调用 docx_v1_document_rawContent 提取文本
4. 对技术设计和测试用例文档重复上述步骤
5. 如果是本地文件（如 .mm 思维导图），直接读取
```

### 步骤 2：生成第 1 部分 - 业务背景

```
1. 从 PRD 提取：
   - 模块目的及解决的问题
   - 业务目标（列表或表格）
   - 目标用户及其操作
2. 输出格式：
   - 概述：2-3 句话
   - 业务目标：表格，包含 目标 | 描述
   - 目标用户：表格，包含 用户类型 | 描述
3. 语言：根据用户要求（中文或英文）
```

### 步骤 3：生成第 2 部分 - 业务流程

```
1. 从 PRD 识别：
   - 入口（页面/路由）
   - 每个页面上的用户操作
   - 状态转换和结果
2. 创建 Mermaid 流程图：
   - 高层用户旅程
   - 页面级流程（首页、品牌页等）
   - 响应式断点
   - 平台差异（PC vs H5）
3. 创建关键操作表格：
   - # | 阶段/区域 | 关键操作 | 描述
4. 可选：生成 PNG 格式的流程图
```

### 步骤 4：生成第 3 部分 - 实现逻辑

```
1. 从技术设计提取：
   - 系统层级（前端、后端、数据层）
   - 组件和模块
   - 模块交互
2. 核心逻辑章节：
   - 数据/分类变更
   - API 行为
   - 各平台展示逻辑
   - 响应式规则
   - 文案和 Toast 文本
3. 边界场景：
   - 列出所有边界情况
   - 格式：触发条件 | 处理方式
4. 使用编号章节：3.1, 3.2.1-3.2.x, 3.3.1-3.3.x
```

### 步骤 5：生成第 4 部分 - API

```
1. API 概述（API Overview）：
   - 简要描述 API 端点及其提供的功能
   - 说明任何第三方集成（如支付网关、短信服务等）
   - 描述 API 的整体架构（REST/GraphQL 等）

2. 端点详情（Endpoint Details）：
   为每个 API 列出以下信息：
   - URL 路径
   - HTTP 方法（GET, POST, PUT, DELETE）
   - 功能用途
   - 请求参数（Request）：JSON 格式示例
   - 响应结果（Response）：JSON 格式示例
   
   示例格式：
   | 端点 | 方法 | 用途 |
   |------|------|------|
   | /api/order/create | POST | 创建新订单 |
   
   请求示例：
   {
     "userId": "1234",
     "items": ["item1", "item2"],
     "total": 45.00
   }
   
   响应示例：
   {
     "orderId": "5678",
     "status": "created"
   }

3. 错误处理（Error Handling）：
   描述 API 可能返回的常见错误及其含义：
   | 状态码 | 错误类型 | 含义 | 处理建议 |
   |--------|----------|------|----------|
   | 400 | Bad Request | 请求参数错误 | 检查必填字段和格式 |
   | 401 | Unauthorized | 未授权访问 | 检查认证 token |
   | 403 | Forbidden | 权限不足 | 确认用户角色权限 |
   | 404 | Not Found | 资源不存在 | 确认 ID 或路径正确 |
   | 500 | Server Error | 服务器内部错误 | 联系后端排查 |

```

### 步骤 6：生成第 5 部分 - 数据结构
```
1. 数据库 Schema：
   - 从技术设计提取表名和用途
   - 列出关键字段和关系
   - 如有则包含 CREATE TABLE 语句
2. 数据校验：
   - 必填/唯一/最大长度规则
   - 业务规则（如 score >= 0）
   - API 输入校验
3. 示例数据：
   - 为每个关键表提供示例行
   - 使用真实但具有说明性的值
```

### 步骤 7：生成第 6 部分 - 测试用例

```
1. 链接到现有测试用例文档
2. 分析测试用例（从思维导图或文档）：
   - 识别测试重点区域
   - 标注已知失败（❌ 标记）
   - 提取 P0 优先级用例
3. 生成测试方法章节：
   - 范围覆盖（按页面/模块）
   - 平台覆盖（PC/H5/APP）
   - 状态覆盖（已收藏、已屏蔽、已售罄等）
   - 视口覆盖（所有断点）
   - 步骤格式：仅限可执行操作
   - 预期结果格式：仅限可观察结果
4. 生成回归清单
```

### 步骤 8：生成第 7 部分 - 典型问题

```
1. 从 PRD、技术设计和测试经验识别：
   - 数据处理问题（状态持久化、导航）
   - UI/UX 问题（响应式、图片压缩）
   - 性能问题（加载时间）
2. 格式：类别 + 描述
3. 不适用的类别标记 N/A
```

### 步骤 9：最终检查和翻译

```
1. 检查所有章节的完整性
2. 确保编号一致（1, 2, 3.1, 3.2.1 等）
3. 如用户需要中文版：
   - 翻译描述性文本
   - 保留英文 UI 文案/Toast 文本不变
   - 保持表格结构一致
4. 保存到 docs/ 文件夹，使用清晰的文件名
```

## 格式规范

### 语言
- 默认：英文
- 按需求：生成中文版，结构不变
- 保留 UI 文案（按钮文本、Toast 消息）的原始语言

### 章节编号
```
1. 业务背景
2. 业务流程
3. 实现逻辑
   3.1 系统架构
   3.2 核心逻辑
      3.2.1 ...
      3.2.2 ...
   3.3 边界场景
      3.3.1 ...
4. API
   4.1 API 概述
   4.2 端点详情
      4.2.1 端点 A
      4.2.2 端点 B
   4.3 错误处理
5. 数据结构
   5.1 数据库 Schema
   5.2 数据校验
   5.3 示例数据
6. 测试用例
7. 典型问题
```

### 表格
- 使用 Markdown 表格带表头
- 对齐列以提高可读性

### 流程图
- 使用 Mermaid 语法
- 按需提供 PNG 导出
- 保持节点文本简洁

### 代码/SQL
- 使用带语言标签的代码块
- 只包含相关片段

## 示例提示词和响应

### 提示词 1：开始生成测试指南
```
用户: 读取这篇PRD文档：[URL] 并生成测试指南
Agent: 
1. 通过 Lark MCP 获取 PRD
2. 如未提供，询问用户技术设计 URL
3. 生成第 1 部分：业务背景
4. 提交审核后再继续
```

### 提示词 2：生成特定章节
```
用户: 现在生成3.2 Core Logic部分
Agent:
1. 从技术设计提取核心逻辑
2. 结构化为编号子章节
3. 包含展示逻辑、API 行为、响应式规则
4. 输出到对话框供审核
```

### 提示词 3：从测试用例生成
```
用户: 根据测试用例生成测试重点和方法
Agent:
1. 读取测试用例文件（思维导图或文档）
2. 识别重点区域、已知问题、P0 用例
3. 生成详细的测试方法章节
4. 包含回归清单
```

### 提示词 4：语言切换
```
用户: 使用英文 / 输出为中文
Agent:
1. 按请求的语言生成或翻译
2. 保留 UI 文案不变
3. 保持结构一致
```

## 文件命名规范

```
{项目名}-test-guide-part{N}-{章节名}.md
{项目名}-test-guide-{语言}.md
```

示例：
- `homepage-redesign-test-guide-part1-business-background.md`
- `homepage-redesign-test-guide-part3-implementation-logic.md`
- `Marketplace-homepage-redesign-Test-Guide-1-中文版.md`

## 使用的 MCP 工具

| 工具 | 用途 |
|------|------|
| `wiki_v2_space_getNode` | 获取飞书 Wiki 节点信息和文档 token |
| `docx_v1_document_rawContent` | 从飞书文档提取文本内容 |
| `Read` | 读取本地文件（如 .mm 思维导图） |
| `Write` | 保存生成的 Markdown 文档 |
| `GenerateImage` | 按需生成 PNG 流程图 |

## 质量检查清单

最终提交前，验证：

- [ ] 全部 7 个章节存在（或标记 N/A）
- [ ] 章节编号一致
- [ ] 表格有表头且对齐正确
- [ ] 流程图覆盖所有入口和关键操作
- [ ] 边界场景列出所有非正常路径
- [ ] 测试方法覆盖所有维度（范围、平台、状态、视口）
- [ ] 突出显示测试用例中的已知问题
- [ ] 全文语言一致（或正确翻译）
- [ ] 文件以清晰的名称保存在 docs/ 文件夹

---

## 注意事项

- 每完成一个章节，提交给用户审核后再继续
- 如果 PRD/技术设计不完整，询问用户进行澄清
- 突出显示测试用例分析中发现的已知失败
- 保持指南对手动测试人员和自动化测试都可操作
- 结构化输出便于 AI Agent 作为 Wiki/上下文使用
