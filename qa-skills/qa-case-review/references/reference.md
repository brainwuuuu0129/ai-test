# 用例评审参考手册

## 1. 问题识别决策树

```
发现用例问题
    │
    ├─ 字段缺失？ ──YES→ 【阻塞级】必须补齐
    │               ↓
    NO            9字段必填（id/title/优先级/类型/执行端/前置/步骤/期望/冒烟）
    │
    ├─ 期望结果模糊？ ──YES→ 【阻塞级】改为可判定
    │                   ↓
    NO                无"正常""成功"等模糊词
    │
    ├─ 前置条件不足？ ──YES→ 【阻塞级】补充4类信息
    │                   ↓
    NO                账号/配置/数据/环境
    │
    ├─ 追溯链路缺失？ ──YES→ 【重要级】补充TP/TI映射
    │                   ↓
    NO                无法追溯到需求或设计
    │
    ├─ 冒烟标记错误？ ──YES→ 【重要级】修正冒烟标记
    │                   ↓
    NO                P0主路径正向才标冒烟
    │
    ├─ 优先级不当？ ──YES→ 【重要级】重新评估优先级
    │                 ↓
    NO              依据不充分或判定错误
    │
    ├─ 标题不清晰？ ──YES→ 【一般级】优化标题描述
    │                 ↓
    NO              标题应体现验证场景
    │
    └─ 其他优化点？ ──YES→ 【一般级】可选改进
                      ↓
                    描述优化/步骤简化等
```

## 2. 修复策略表

### 2.1 阻塞级问题修复

| 问题类型 | 典型表现 | 标准修复方法 | 示例 |
|---------|---------|-------------|------|
| **字段缺失** | 缺id/优先级/执行端等 | 根据用例内容补充缺失字段 | 缺"执行端" → 补充Web/App/API |
| **期望模糊** | "正常显示""验证成功" | 改为具体验证点：状态码/UI文本/字段值/数据变化 | "登录成功" → "接口返回200+跳转/home+显示'张三'" <br> "支付成功" → "接口200+订单状态已支付+余额-100+生成流水" |
| **前置不足** | 仅"有账号""已登录" | 补充4类信息：账号权限/配置/数据/环境 | "有账号" → "test@example.com/Pass123 + user_id=10001" |
| **步骤不清** | "登录并提交" | 拆分为可复现的具体步骤 | "登录并提交" → "1.打开登录页 2.输入账号密码 3.点击登录" |
| **无法执行** | 依赖信息缺失 | 标注【未澄清】，进入待澄清清单 | 【未澄清：第三方支付账号｜影响：TC-010｜需要：开发提供测试账号】 |

### 2.2 重要级问题修复

| 问题类型 | 典型表现 | 标准修复方法 | 示例 |
|---------|---------|-------------|------|
| **追溯缺失** | 无TP/TI映射 | 补充来源，建立追溯链路 | TC-005 → 补充"来源：TP-005" |
| **冒烟错误** | 异常用例标冒烟 | 严格执行P0∩主路径正向规则 | TC-008（密码错误）标冒烟 → 改为"否"<br>P0异常场景不标冒烟，仅P0主路径正向标冒烟 |
| **优先级不当** | P0判定不合理 | 重新评估，给出判定依据 | TC-020（边界场景）标P0 → 改为P2 |
| **类型错误** | 类型与场景不符 | 按实际场景重新分类 | "功能-正向"实际是异常 → 改为"异常-参数错误" |
| **重复用例** | 多个用例验证相同点 | 合并为参数化用例，或明确差异点 | TC-001/TC-002都验证正常登录 → 合并<br>跨端验证可合并为混合执行端 |

### 2.3 一般级问题修复

| 问题类型 | 典型表现 | 标准修复方法 | 示例 |
|---------|---------|-------------|------|
| **标题不清** | "登录测试""用户测试" | 明确验证场景和期望 | "登录测试" → "正确用户名密码登录成功" |
| **描述冗余** | 过长/重复信息 | 精简，保留关键信息 | 删除重复的背景说明 |
| **步骤冗长** | 10+步骤 | 合并相关步骤，或拆分为多个用例 | 10步骤 → 拆为2个用例 |

## 3. 回归集规则

**P0必回归**：
- 所有P0用例必须进入回归集（100%覆盖）
- 依据：核心链路/资金安全/数据安全/合规

**P1推荐回归**：
- 变更影响范围内的用例（影响分析）
- 高频路径用例（用户使用频率）
- 历史缺陷相关用例（风险评估）

**回归集依据**：必须给出变更影响分析/风险评估/历史缺陷

**示例**：
```
P0必回归（10条）：
- TC-001~TC-005：登录主流程（核心链路）
- TC-020~TC-025：支付流程（资金安全）

P1推荐回归（15条）：
- TC-030~TC-035：订单查询（本次变更影响订单模块）
- TC-050~TC-055：用户中心（高频路径，DAU占比60%）
- TC-080~TC-085：密码修改（历史缺陷：上次发布引入bug）
```

## 4. 常见问题FAQ

| 问题 | 检测方式 | 处理方式 | 示例 |
|------|---------|---------|------|
| **缺少PRD/TD** | 用例无追溯源 | 重点评审可执行性，覆盖性评审降级为建议项 | 标注"无PRD，无法验证覆盖性" |
| **冒烟集超标** | 冒烟比例>15% | 严格执行P0∩主路径正向，排除异常/边界/权限/并发 | 重新评估冒烟标记 |
| **重复用例** | 多个用例验证同一点 | 合并为参数化；跨端验证合并为混合执行端 | TC-001/TC-002都验证正常登录 → 合并 |
| **不可测用例** | 依赖人工/第三方 | 依赖人工判断→探索式测试；依赖第三方→Mock或手工验证 | Mock支付网关 |
| **优先级争议** | P0/P1判定不一致 | 参考core-rules.md优先级判定规则，资金/安全/合规→P0 | 支付必P0 |
| **用例粒度不当** | 过粗或过细 | 一个TC验证一个具体场景，3-10步为宜 | 过粗拆分，过细合并 |
