# 一键生成用例参考手册

## 1. 三条路径详细流程

### 1.1 路径选择策略

| 路径 | 输入特征 | 执行阶段 | 适用场景 | 优缺点 |
|------|---------|---------|---------|--------|
| **路径A** | 原始PRD+TD | 1→2→3→4→5 | 从零开始完整分析 | ✓完整覆盖 / ✗耗时较长 |
| **路径B** | 已有TP/TI清单(≥3个) | 跳过1/2，3→4→5 | 复用已有分析⭐推荐 | ✓快速复用 / ⚠依赖TP/TI质量 |
| **路径C** | 仅PRD(无TD) | 1→3→4→5（阶段3仅对TP排序，无TI合并） | 快速启动⭐常用 | ✓快速启动 / ⚠技术验证不完整 |

**内部执行阶段**（对应SKILL.md步骤2「执行分析」的展开）：PRD分析 → TD分析 → 合并排序 → 用例生成 → 质量输出

**路径A合并规则**：同模块下 TP 用例先、TI 用例后；同一业务功能的 TI 归入对应 function，不单独成模块；同一功能点既有 TP 又有 TI 时，TI 作为该 test_point 下的独立 cases 追加。

### 1.2 路径差异处理

**路径B特殊处理**：
- 输入TP/TI质量校验：检查是否包含id/name/source/priority
- 不合格项标注【待修正】，并在待澄清清单列出具体问题
- 示例：`TP-005缺少source字段 → 标注【待修正：缺少追溯源】`

**路径C特殊处理**：
- 技术类验证点标注【缺少TD】
- 额外输出《技术验证待补充清单》，模板如下：

```
## 技术验证待补充清单（路径C缺少TD）

| 验证点 | 缺失信息 | 影响TC | 需要 |
|--------|---------|-------|------|
| 幂等机制验证 | 【缺少TD：幂等键/实现方式/有效期/并发策略】 | TC-幂等-xxx | 开发提供幂等设计方案 |
| 重试补偿验证 | 【缺少TD：重试次数/间隔/副作用处理】 | TC-重试-xxx | 开发提供重试策略 |
| 接口鉴权验证 | 【缺少TD：token结构/有效期/刷新机制】 | TC-鉴权-xxx | 开发提供接口安全设计 |
```
（按实际缺失的技术点逐行填写）

## 2. JSON格式规范

**四层结构**：modules → functions → test_points → cases

**9个必填字段**：id, title, 优先级, 用例类型, 执行端, 前置条件, 测试步骤, 期望结果, 冒烟标记

**特别要求**：
- `前置条件`：必须包含4类信息（账号权限/配置开关/数据准备/环境依赖）
- `期望结果`：必须可判定（状态码/字段值/UI文本/数据变化）
- `冒烟标记`：P0 ∩ 主路径正向流程

## 3. RTM追溯矩阵

**双向追溯**：
- 正向：REQ/TD → TP/TI → TC
- 反向：TC → TP/TI → REQ/TD

**孤儿检查**：
- 孤儿REQ/TD → 标注【未覆盖】
- 孤儿TP/TI → 标注【待补充用例】
- 孤儿TC → 标注【缺少追溯源】

## 4. 质量门槛与自检机制

### 4.1 质量门槛

| 维度 | 目标值 | 路径C调整 |
|-----|--------|----------|
| 需求覆盖率 | ≥95% | REQ→TP→TC即可 |
| 风险覆盖率 | 100%适用项 | 技术类风险标注【缺少TD】 |
| 异常用例占比 | 20-35% | 技术异常可标注【缺少TD】 |
| 边界值覆盖率 | 关键字段≥90% | 数值/长度/容量边界±1 |
| 状态流转覆盖 | 100%合法转换+主要非法转换 | 无状态功能可跳过 |
| 权限角色覆盖 | 核心操作×角色≥80% | 无权限控制可跳过 |
| 用例完整率（9字段） | 100% | 无差异 |
| 冒烟集比例 | 5-15% | 无差异 |

### 4.2 自检评分细则（≥90分可交付）

**覆盖率维度（50分）**

| 检查项 | 满分 | 扣分规则 | 示例 |
|--------|------|---------|------|
| 需求覆盖率 | 15 | 每低5%扣2分 | 覆盖率90% → 扣2分，得13分 |
| 风险场景覆盖 | 8 | 每缺失1项扣2分（上限8分） | 5类风险缺失并发验证 → 扣2分，得6分 |
| 异常用例占比 | 7 | <15%或>40%扣3分；<10%或>45%扣7分 | 异常占比12% → 扣3分，得4分 |
| 边界值覆盖 | 8 | 每缺失1个关键字段边界扣2分 | 金额/数量字段无边界验证 → 扣4分，得4分 |
| 状态流转覆盖 | 7 | 缺失合法转换每条扣2分；无非法验证扣3分 | 订单状态缺"已发货→已完成" → 扣2分，得5分 |
| 权限角色覆盖 | 5 | 核心操作缺失角色验证每处扣1分 | 删除操作未验证普通用户越权 → 扣1分，得4分 |

**质量维度（30分）**

| 检查项 | 满分 | 扣分规则 | 示例 |
|--------|------|---------|------|
| 9字段完整率 | 15 | 每缺失1个TC扣1分（上限15分） | TC-003/TC-008缺"执行端" → 扣2分，得13分 |
| 期望多层覆盖 | 10 | 每个模糊表述扣2分；写操作仅验响应码未覆盖数据层每处扣1分（上限10分） | TC-005用"正常显示" → 扣2分；TC-010下单仅验HTTP200无DB验证 → 扣1分 |
| 前置条件完整 | 5 | 每个TC缺失1类信息扣0.5分 | TC-010缺"配置开关" → 扣0.5分，得4.5分 |

**追溯维度（20分）**

| 检查项 | 满分 | 扣分规则 | 示例 |
|--------|------|---------|------|
| 无孤儿REQ/TD | 7 | 每个孤儿扣2分（上限7分） | REQ-005未覆盖 → 扣2分，得5分 |
| 无孤儿TP/TI | 7 | 每个孤儿扣2分（上限7分） | TP-012无对应TC → 扣2分，得5分 |
| 冒烟规则正确 | 6 | 每个错误项扣2分（上限6分） | TC-020是异常用例但标冒烟 → 扣2分，得4分 |

**扣分上限**：每项扣分不超过该项满分，总分不低于0分

### 4.4 修复循环规则

评分 < 90 时不直接交付，启动修复循环（最多2轮）后重新执行自检。

**循环逻辑**：
```
自检评分
  ≥ 90 → 进入步骤5，正常输出
  < 90 → 第1轮修复（自动修复所有可修复问题）→ 重新自检
           ≥ 90 → 进入步骤5
           < 90 → 第2轮修复（再次修复残余可修复问题）→ 重新自检
                    ≥ 90 → 进入步骤5
                    < 90 → 进入步骤5，自检报告注明"已完成2轮修复"，待澄清列出剩余无法自动修复的问题
```

**可自动修复（AI直接修复，无需用户介入）**：

| 问题类型 | 自动修复方式 |
|---------|------------|
| 字段缺失（执行端/优先级/用例类型等） | 根据用例上下文推断补全 |
| 期望结果含模糊词（"正常""成功""符合预期"） | 改写为状态码/字段值/UI文本等可判定描述 |
| 写操作期望仅验响应码（缺数据层） | 补充DB/缓存/状态字段的验证期望 |
| 冒烟标记错误（异常/边界/权限/并发用例标"是"） | 修正为"否" |
| 测试步骤含占位符（"输入数据"/"调用接口"等） | 根据前置条件和测试场景补充具体值 |
| 前置条件缺类（4类不全） | 根据用例内容补全缺失类别 |
| title笼统（"验证...功能"/"测试...流程"） | 按§7.1规范改写为"动作+条件→期望摘要"格式 |

**不可自动修复（加入待澄清，等待用户介入）**：

| 问题类型 | 待澄清标注 |
|---------|----------|
| 孤儿REQ/TD（需求无法覆盖） | 【待澄清：REQ-xxx内容缺失，无法生成TC｜需要：提供该需求详情】 |
| 期望文案/错误码PRD未明确 | 【未澄清：具体文案/错误码｜需要：产品确认后补充】 |
| 缺少TD导致技术断言无法填写 | 【缺少TD：技术验证点无法补充断言｜需要：开发提供TD】 |
| 需求覆盖率 < 80%（输入严重不足） | 【输入不完整：PRD覆盖不足，建议补充后重新生成】 |

**自检报告头部**在修复轮次中须注明当前状态：
```
当前为第N轮修复后自检（本轮修复了X处问题，剩余Y处需用户介入）
```

## 6. TP→TC 系统展开矩阵

**用法**：展开每个 TP/TI 前，逐行扫描下表，勾选适用维度，每个命中维度至少产出1条TC。未适用的维度需在注释中说明跳过原因。

| 展开维度 | 展开场景 | 适用条件 | 最少TC数 |
|---------|---------|---------|---------|
| **正向基准** | 主流程成功路径（冒烟候选） | 必须 | 1条 |
| **参数变体** | 关键参数的不同有效值（多角色/多状态/多渠道） | 存在多个有效等价类时 | 每种变体1条 |
| **异常-参数错误** | 必填缺失/格式错误/超长/非法字符/类型错误 | 存在输入校验时 | ≥2条 |
| **异常-状态不满足** | 前置状态不符（如已支付订单重复支付/已关闭订单核销） | 存在状态机时 | 每条非法流转1条 |
| **异常-依赖失败** | 外部服务超时/熔断/降级/返回错误码 | 存在外部依赖时 | ≥1条 |
| **边界值** | 最小值/最大值/边界±1/等于阈值/特殊值(0/负数/空/超长) | 存在数值/长度/数量约束时 | 每个关键字段≥3条 |
| **等价类-无效** | 不同类型的无效输入（空值/非法格式/越界/SQL注入/XSS） | 存在多种无效输入类型时 | 每种无效类型1条 |
| **权限** | 未登录/低权限角色/横向越权（他人数据）/纵向越权（高权限操作） | 存在权限控制时 | ≥2条 |
| **并发/幂等** | 同资源并发提交/重复请求/超时重试/断网重发 | 写操作/资金/状态变更时 | ≥1条 |
| **兼容** | 多端差异（Web/App/API）/弱网/版本兼容/降级开关 | 多端功能/网络敏感操作时 | 按端数或降级场景数 |
| **可观测性** | 关键操作日志写入/埋点触发/告警规则验证 | TI来自TD可观测性要求时 | ≥1条 |

**展开示例**（TP：验证优惠券核销）：

| 维度 | TC标题示例 |
|------|-----------|
| 正向基准 | 满足门槛金额使用有效券下单→核销成功、订单折扣正确 |
| 参数变体 | 满减券/折扣券/免运费券各类型正向核销 |
| 异常-参数错误 | couponId为空→返回400+错误码COUPON_REQUIRED |
| 异常-状态不满足 | 使用已过期优惠券→返回错误码COUPON_EXPIRED |
| 异常-状态不满足 | 使用他人优惠券→返回错误码COUPON_NOT_OWNED |
| 异常-依赖失败 | 优惠券服务超时→订单创建失败且无核销记录 |
| 边界值 | 订单金额=门槛值时可用；金额=门槛-0.01时不可用 |
| 等价类-无效 | couponId传入SQL注入字符串→返回400，无SQL执行 |
| 权限 | 未登录用户调用核销接口→返回401 |
| 并发/幂等 | 同一张券并发提交2次→仅1次成功、无重复核销 |

## 7. 用例内容质量规范

### 7.1 用例标题（title）命名规范

**格式**：`【执行动作/场景特征】+【关键条件/数据特征】→【期望结果摘要】`

| 质量 | 示例 | 问题 |
|------|------|------|
| ✅ 推荐 | 使用过期优惠券下单→提示"券已失效"且订单未创建 | 场景+条件+期望，一目了然 |
| ✅ 推荐 | 并发提交同一优惠券两次→仅1次成功核销 | 并发场景和期望清晰 |
| ✅ 推荐 | 金额=门槛值-0.01时使用满减券→提示"未满足使用条件" | 边界条件精确描述 |
| ❌ 禁止 | 验证优惠券功能 | 太笼统，无场景无期望 |
| ❌ 禁止 | 测试正常登录 | "正常"语义模糊 |
| ❌ 禁止 | 用户使用有效邮箱密码在登录页面完成登录操作 | 过长且无期望摘要 |

**各类用例命名要点**：
- 正向用例：体现关键输入特征（"有效邮箱密码"而非"正确信息"）
- 异常用例：体现异常类型（"过期优惠券"而非"无效券"）
- 边界用例：写出具体边界值（"金额=0.01元"而非"最小金额"）
- 并发用例：含"并发"/"重复"/"同时"等特征词

### 7.2 测试步骤书写规范

**核心原则**：1步=1个操作；包含具体值而非占位描述；可直接执行无需二次理解。具体值**优先从 PRD/TD 原文摘录**实际字段名、ID、枚举值（如接口示例中的 kitchenId=12815、status='Active'），禁止使用"某产品"/"某账号"/"Active产品"等通用占位符。

| 类型 | 推荐写法 | 禁止写法 |
|------|---------|---------|
| UI操作 | 在金额输入框输入"-1"，点击"确认"按钮 | 输入金额，提交 |
| API调用 | POST /api/orders，body={"couponId":"CPN001","amount":100} | 调用下单接口 |
| DB验证 | 查询 order 表 WHERE order_id='O001'，检查 status 字段值 | 检查数据库 |
| 登录前置 | 使用账号 test@example.com / Pass123456 登录系统 | 登录系统 |

**步骤数量参考**：
- 简单操作（单接口/单页面）：3-5步
- 复杂流程（多步骤流程）：6-10步（超过10步考虑拆分TC）
- API用例：步骤=请求构造+验证（最少2步）

### 7.3 期望结果分层框架

**核心要求**：期望结果需覆盖"响应层 + 数据层 + 副作用层"，写操作用例不得只验响应层。

| 执行端 | 响应层（必验） | 数据层（写操作必验） | 副作用层（有则验） |
|-------|-------------|------------------|-----------------|
| **API** | HTTP状态码 + 响应body关键字段 | DB表字段变更 / 缓存更新 | 消息队列事件 / 三方回调触发 |
| **Web** | 页面跳转URL / 元素出现消失 / Toast文本 | 刷新后数据持久化 | 服务端接口调用结果 |
| **App** | 界面元素变化 / 弹窗文本 | 本地存储/缓存更新 | 服务端数据变化 |
| **Job/异步** | 任务执行日志（批次ID+处理数量+耗时） | DB记录变化 / 状态字段更新 / order_id回填 | 通知/消息是否发送；幂等验证（重复触发→记录数不变）|

**示例对比**：

| 质量 | 期望结果写法 |
|------|------------|
| ❌ 不完整 | 接口返回200（只验响应码，业务结果未验） |
| ⚠️ 部分 | 接口返回200，返回orderId（缺数据层：DB是否写入）|
| ✅ 完整 | 1. HTTP 200，body.code=0，body.data.orderId非空 / 2. order表新增记录，status=PENDING，amount=100 / 3. 库存表sku_stock字段-1 |

**不可验证时**：
- 三方副作用无法直接断言 → 标注【仅监控：通过日志/告警验证，非主动断言】
- UI文案未确认 → 标注【未澄清：具体文案｜需要：产品提供】

## 8. 常见问题处理

### 8.1 输入相关问题

| 问题 | 检测方式 | 处理方式 | 示例 |
|-----|---------|---------|------|
| **路径B输入TP/TI质量不足** | 检查是否包含id/name/source/priority | 不合格项标注【待修正】，在待澄清清单列出 | `TP-005缺source → 【待修正：无法追溯到需求】` |
| **PRD内容过于简略** | 检查是否包含功能描述/用户角色/流程 | 标注【输入不完整】，输出有限用例+详细待澄清清单 | `缺少用户角色定义 → 无法判定权限测试范围` |
| **混合输入格式混乱** | 同时存在PRD+部分TP | 优先路径A完整分析，提示"检测到已有TP-xxx，将合并" | - |

### 8.2 生成相关问题

| 问题 | 检测方式 | 处理方式 | 示例 |
|-----|---------|---------|------|
| **异常用例占比不达标** | 自检时异常占比<20%或>35% | 在自检报告标注【异常场景不足/过多】，不阻塞生成 | 占比12% → 建议补充参数错误/依赖异常等场景 |
| **JSON用例量较大** | 用例总数>50 | 直接完整写入文件，无截断问题；窗口仅输出摘要+自检+待澄清 | 用例80条 → 完整写入test_cases/xxx.json |
| **冒烟集超标** | 自检时冒烟比例>15% | 严格复核P0且主路径正向（排除异常/边界/权限/并发） | 冒烟25% → 将异常用例TC-008/TC-015改为"否" |
| **期望结果难以可判定** | PRD无明确错误码/UI文案 | 标注【未澄清：具体判定标准】，不编造内容 | `【未澄清：密码错误提示文案｜影响：TC-002｜需要：产品提供文案】` |

### 8.3 路径特殊问题
