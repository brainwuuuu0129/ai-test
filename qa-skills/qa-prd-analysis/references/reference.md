# PRD 分析参考手册

## 1. 输入处理流程

### 1.1 PRD类型决策树

```
用户输入
    │
    ├─ 完整PRD文档（含需求/流程/规则）？ ──YES→ 标准流程：事实摘录 → 分析 → TP清单
    │                                          ↓
    NO                                      按10章节输出完整分析报告
    │
    ├─ 原型图为主（少量文字说明）？ ──YES→ 视觉流程：提取UI/流程 → 标注不清晰项
    │                                     ↓
    NO                                  明确分析局限性，建议补充文字说明
    │
    ├─ 口头需求（会议纪要/聊天记录）？ ──YES→ 最小流程：抽取关键点 → 大量【未澄清】
    │                                       ↓
    NO                                    输出最小澄清清单，先明确需求再分析
    │
    └─ 格式无法识别 → 询问用户：是完整PRD/原型图/口头需求？
```

### 1.2 三步处理法

1. **先形成事实摘录**：从PRD中抽取可引用事实（关键段落/字段/流程/规则）
2. **再做推导分析**：基于事实摘录产出测试化分析
3. **无法识别处理**：使用【未澄清】格式

### 1.3 关键信息缺失处理

对以下信息缺失，先列"最小澄清清单"，但不阻塞已确定部分的分析：
- 范围（In/Out）
- 角色/权限/状态机
- 异常处理/错误码/降级策略
- 数据口径/字段校验
- 依赖系统/第三方集成
- 灰度发布/回滚策略
- 监控告警/可观测性

### 1.4 视觉内容处理

从UI截图/原型图/流程图/状态机/时序图提取关键要点。不清晰部分标注【未澄清】。

## 2. TP（测试点）产出规范

### 2.1 TP定义与追溯

**TP定义**：来源于PRD，验证业务行为/规则/用户体验  
**追溯链路**：REQ-xxx → TP-xxx → TC-xxx  
**注意**：PRD分析只产出TP，不产出TI（TI由TD分析产出）

### 2.2 TP拆分粒度指南

| 粒度 | 示例 | 问题 | 建议 |
|------|------|------|------|
| **过粗** | TP-001：验证用户登录 | 无法明确验证范围，易遗漏场景 | 按登录方式拆分（邮箱/手机号/第三方） |
| **合适** | TP-001：验证邮箱格式用户名登录<br>TP-002：验证手机号格式用户名登录 | 验证范围明确，可展开为具体用例 | ✓ 推荐粒度 |
| **过细** | TP-001：验证邮箱用户名登录成功<br>TP-002：验证邮箱用户名密码错误<br>TP-003：验证邮箱用户名账号锁定 | TP与TC界限模糊，管理成本高 | 合并为TP-001，正向/异常作为TC |

**拆分原则**：
- 按**功能点/业务规则**拆分，不按**正向/异常**拆分（正向/异常是TC层面）
- 一个TP可展开为3-10个TC（含正向/异常/边界）
- TP名称应体现"验证什么"，而非"怎么验证"

## 3. 风险识别与5类场景

### 3.1 风险结构

每个风险（RISK-xxx）必须包含：
- **原因**：为什么存在这个风险
- **影响**：风险发生后的影响范围和严重程度
- **建议验证方式**：如何验证（测试用例/监控验收/压测/审计等）

### 3.2 五类风险识别表

| 风险类别 | 典型场景 | 识别方法 | 验证重点 |
|---------|---------|---------|---------|
| **功能** | 主流程不可用/关键分支缺失/状态流转异常 | 检查流程完整性、状态机覆盖度 | 主流程+关键分支+边界+异常 |
| **权限安全** | 未鉴权/越权访问/敏感数据泄露/注入攻击 | 检查鉴权规则、角色权限、数据脱敏 | 鉴权/越权(横向纵向)/数据脱敏/注入防护 |
| **数据可靠性** | 重复提交/并发冲突/数据丢失/状态不一致 | 检查幂等/事务/并发处理机制 | 幂等/重复提交/并发写/超时重试 |
| **兼容发布** | 多端差异/版本不兼容/灰度异常/降级失败 | 检查多端适配、版本兼容策略、灰度方案 | 多端一致性/弱网/灰度/回滚/降级 |
| **可观测性** | 关键操作无日志/异常无告警/链路断点 | 检查日志埋点、指标采集、告警配置 | 日志/埋点/告警/链路追踪 |

**识别技巧**：
- 从PRD的"异常处理""非功能需求""依赖系统"章节重点挖掘
- 对于"未明确说明"的场景，标注【未澄清】并进入待澄清清单
- 资金/数据安全/合规类风险优先级直接判定为P0

## 4. 特殊场景与边界处理

| 场景 | 检测方式 | 处理方式 | 示例 |
|------|---------|---------|------|
| **多PRD合并** | 同时收到多个PRD文档 | REQ编号增加前缀区分来源 | REQ-USER-001, REQ-ORDER-001 |
| **增量需求** | PRD标注"v2.0""补充需求" | 标注[新增]/[修订]，给出回归建议 | TP-010 [新增]，TC-001 [修订] |
| **版本迭代** | PRD包含变更说明 | 标注[新增]/[修改]/[删除]，对比v1.0 | TP-005 [修改]：3次→5次 |
| **原型图为主** | 只有Figma/Axure文件 | 按视觉规则提取，明确分析局限性，建议补充文字说明 | 标注"原型图未标明此处逻辑"，列入待澄清 |
| **口头需求** | 会议纪要/聊天记录 | 最小事实摘录+最小澄清清单（P0必须澄清/P1建议澄清） | 先输出待澄清清单，再分析已明确部分 |

## 5. 常见问题处理

| 问题 | 检测方式 | 处理方式 | 示例 |
|------|---------|---------|------|
| **PRD信息不完整** | 关键章节缺失/规则不明确 | 不阻塞已确定部分，对缺失标注【未澄清】，给出最小澄清清单 | 见§4特殊场景处理 |
| **PRD有矛盾冲突** | 不同章节描述不一致 | 指出矛盾点，标注影响，要求产品确认，不做假设 | "§1说3次锁定，§2说5次锁定" |
| **仅有原型图** | 只有Figma/Axure文件 | 按视觉规则提取，明确分析局限性，建议补充文字说明 | 见§4特殊场景处理 |
| **冒烟标记判断** | 不确定是否标冒烟 | 决策树：P0？→主流程？→正向流程？→三者都是才是冒烟 | 异常用例不标冒烟 |
| **TP粒度不确定** | 不确定拆分粒度 | 参考§2.2拆分粒度指南 | 按功能点拆分，不按正向/异常拆分 |
| **5类风险识别不全** | 只识别功能风险 | 参考§3.2五类风险表，逐类检查 | 权限/数据/兼容/可观测也要覆盖 |
