# TD 分析（面向测试）

## 适用场景

用户提供TD/技术方案/接口说明，需要产出技术验证策略、风险分析和TI清单

## 规则依赖

开始分析前读取 `.cursor/skills/_shared/core-rules.md`，重点关注 §1 §3 §5 §6

## 执行流程（AI必读）

### Step 1: 链路抽取
组件图 → 依赖关系 → 时序流程 → 每段链路标注可验证手段

### Step 2: 可靠性分析
检查5要素（事务/幂等/重试/补偿/并发）：
- 明确策略 → 设计验证TI
- 策略缺失 → 标注【未澄清】
- **逐模块扫描TD模式**：识别 Job / 异步 / 枚举变更 / 强制校验 / 状态机 / 计算逻辑 / 改动现有接口 / RBAC权限，按 [reference.md §7](references/reference.md#7-常见td模式检查表) 检查表为每个命中模式追加 TI

### Step 3: 可观测性检查
检查4要素（日志/指标/告警/链路）：
- 关键操作有日志？指标可采集？异常有告警？链路可追踪？
- 不足 → 说明不可测原因或建议补充

### Step 4: TI清单产出
逐条产出TI-xxx，每个TI包含：
- id/name/source/priority
- 验证手段（接口/日志/DB/指标/链路追踪/Mock）
- 环境要求（测试环境/预发/生产只读）
- 通过标准（可判定，禁止"验证成功""正常"）

### Step 5: 质量检查
- ✓ 每条TI通过标准可判定？（禁止"成功""正常"等模糊词）
- ✓ 可靠性5要素全部明确或标注【未澄清】？
- ✓ 可观测性4要素覆盖或说明不可测原因？
- ✓ 每个命中TD模式都已追加对应TI？
- ✓ 每条TI有验证手段和环境要求？

> 一致性策略表、可观测性清单、可测性判定简表见 [reference.md §1-2](references/reference.md#1-一致性与可靠性策略)；常见TD模式检查表见 [reference.md §7](references/reference.md#7-常见td模式检查表)

## 快速口诀

| 维度 | 口诀 |
|------|------|
| **可靠性5要素** | 事务/幂等/重试/补偿/并发（缺失标【未澄清】） |
| **TD模式扫描** | Job/异步/枚举变更/强制校验/状态机/计算逻辑/改动现有接口/RBAC权限（逐模块识别，命中必追加TI） |
| **可观测性4要素** | 日志/指标/告警/链路（不足说明不可测） |
| **TI必含** | 验证手段+环境要求+通过标准（可判定） |
| **TI追溯** | TD → TI → TC，来源于技术设计 |
| **不做假设** | 幂等键/重试策略/错误码/日志格式未明确→【未澄清】 |

**质量门槛**：
- ✓ 链路清晰，每段有可验证手段
- ✓ 每个TI通过标准可判定（禁止"成功""正常"）
- ✓ 可靠性5要素明确或标注【未澄清】

> 优先级判定、TI字段规范详见 [core-rules.md](../_shared/core-rules.md)

## 输出结构

按以下8个章节输出技术分析报告：

1. **架构与边界** - 组件/服务/依赖系统/关键链路
2. **接口契约** - 鉴权/入参出参/错误码/限流熔断重试
3. **数据流与一致性** - 事务边界/幂等/缓存/补偿回滚/并发冲突
4. **配置与发布** - 开关/配置中心/兼容策略
5. **可观测性** - 日志/指标/告警阈值/审计日志
6. **风险清单** - RISK-xxx（原因→影响→验证方式）
7. **测试切入点清单** - TI-xxx（验证手段/环境/通过标准）
8. **待澄清清单** - 【未澄清】标记+Q-xxx问题+QA回答区

## 核心示例

**输入TD片段**：
```
支付接口幂等设计：
- 幂等键：order_id
- 实现方式：Redis SET NX，过期时间24小时
- 重复请求：返回原支付URL
```

**Step 2: 可靠性分析**

| 策略 | TD描述 | 验证要点 |
|------|--------|---------|
| 幂等 | Redis SET NX + order_id | 重复请求仅执行一次 |
| 并发 | 【未澄清：并发控制方式】 | 并发下幂等有效 |

**Step 4: TI清单产出**

| TI-ID | 验证点 | 来源 | 优先级 | 验证手段 | 环境要求 | 通过标准 |
|-------|-------|------|--------|----------|---------|---------|
| TI-001 | 验证幂等机制 | TD-001 §幂等设计 | P0 | 接口调用+Redis查询 | 测试环境 | 同一order_id重复请求仅创建一次 |
| TI-002 | 验证幂等键过期 | TD-001 §幂等设计 | P1 | 接口+时间控制 | 测试环境 | 24h后同order_id可再创建 |
| TI-003 | 验证并发幂等性 | TD-001 §幂等设计 | P0 | 并发压测100 | 测试环境 | 同order_id仅1个成功 |

**待澄清示例**：
```
【未澄清：支付流水写入和幂等键设置是否在同一事务｜影响：TI-003｜需要：开发确认事务边界】
```
